## 一致性审计结论：不是“完全一致”，而是“核心一致、执行层有衰减”

作为产品形态设计文档，它准确地继承了上一轮“底层统一 + 上层场景化”的核心结论，整体方向正确。但如果用“细胞级”标准逐条对照，会发现**决策依据、边界规则、落地细节、与当前实现的对接**四个层面有明显衰减。

我把它分成三档问题：

| 级别 | 问题 | 影响 |
|------|------|------|
| **高** | 缺少“三种方案对比”与决策依据 | 团队/评审者不知道“为什么不独立、为什么不合并”，结论变成断言 |
| **高** | 权限冲突规则表述模糊 | “不互相覆盖”无法指导工程实现和 UX 文案 |
| **中** | 缺少“当前实现 → 目标形态”的 gap 分析 | 文档停留在理想设计，无法直接驱动排期和迭代 |
| **中** | 缺少 UI/文案级具体示例 | 产品、设计、开发对同一段话理解可能不一致 |
| **低** | 个别表述过于绝对、指标缺乏 baseline | 降低文档的可信度和可执行性 |

## 作为高级产品总监的优化建议

### 一、结构层：补齐“决策依据”和“现状 gap”

#### 1. 必须补回「三种方案对比」章节
当前文档直接抛出结论，评审者会问：“为什么不能让数据室完全独立做一个 SKU？”  
建议在 §2 之后新增 **§3 方案对比与决策依据**，沿用上一轮的 A/B/C 三方案，并给出评分维度：

| 维度 | A. 完全独立 | B. 底层统一+上层场景化（推荐） | C. 全部围绕数据室 |
|------|-------------|-------------------------------|-------------------|
| 用户激活摩擦 | 中 | 低 | 高 |
| 工程成本 | 高（两套解析/AI/权限） | 中 | 中 |
| 资产复用 | 差 | 优 | 中 |
| AI/分析一致性 | 差 | 优 | 优 |
| 商业化 upsell | 弱 | 强 | 中 |
| 数据安全/合规 | 需两套审计 | 统一审计 | 统一审计 |

#### 2. 新增「当前实现 vs 目标形态差异」章节
现有 PRD/TDD 已经存在 `deal_room_documents` 关联表，说明数据层大概率已经是“引用关系”。但前端/交互上可能存在“Room 内上传的文件不进入文档库”的实现惯性。  
建议增加一节，列出需要验证/对齐的点：

- 数据室详情页的上传按钮，后端是否走的还是 `POST /documents`？
- Room 创建向导里的上传，是否先写 documents 表再写 deal_room_documents？
- 当前 Dashboard 是否已经把 Link 访问和 Room 访问合并成同一个 Signal？
- 文档详情页是否已有“暴露面”组件，还是只是一个 Tab？

### 二、内容层：把抽象原则变成可执行规则

#### 3. 权限冲突必须给出「决策矩阵」
“权限不互相覆盖”这句话太轻。建议改为表格 + 用户可见文案：

| 场景 | 规则 | 用户-facing 说明 |
|------|------|------------------|
| 文档已通过 Link 公开分享，又被加入 Room | Link 仍然有效；Room 成员按 Room 权限访问 | “此文档已通过 1 个公开链接共享，加入数据室不会自动撤销该链接。” |
| 用户被移出 Room，但曾收到 Link | 仍可访问 Link（除非 Link 单独被撤销） | “您已被移出该数据室，但文档所有者发送给您的链接仍可访问。” |
| Room 内文件夹权限限制某成员访问某文件夹 | 该成员在 Room 内看不到该文件夹；不影响其通过 Link 看完整文档 | “该成员无法在此数据室中查看‘财务’文件夹。” |
| 文档 Owner 在 Library 中撤销 Link | 所有通过该 Link 的访问失效，Room 成员不受影响 | “公开链接已撤销，数据室成员仍可正常访问。” |

#### 4. 生命周期边界要给出「决策树」
尤其是文档版本更新、归档、删除三个动作对 Room 和 Link 的影响，建议用流程图或决策树表达，避免工程/产品理解不一致。

示例：

```
文档库中更新文件
├── 未被任何 Room 锁定版本
│   └── 所有 Room 自动展示最新版
├── 某 Room 已锁定版本
│   └── 该 Room 保持锁定版本，其他 Room 跟随最新版
└── 历史版本
    └── 保留，便于审计与回滚
```

```
文档库中删除文件
├── 文件未被任何 Link/Room 引用
│   └── 允许硬删除
├── 文件被 Link/Room 引用
│   ├── 默认：归档（Link/Room 仍可访问）
│   └── 可选：先从所有引用中移除，再删除
└── 历史版本保留策略：保留 90 天
```

#### 5. 增加 UI/文案级示例
产品文档最容易出偏差的地方，就是大家把同一段中文理解成不同的界面。建议补充：

**文档详情页 - 暴露面组件示例：**

```text
共享暴露面
├── 公开链接（1）
│   └── 创建于 2026-06-20 · 已查看 12 次 · [撤销]
├── 受限链接（2）
│   ├── 投资人尽调链接 · [管理]
│   └── LP 报告链接 · [管理]
└── 数据室（1）
    └── Series A Due Diligence · [查看数据室]
```

**“升级为数据室”入口示例：**

```text
[此文档已被查看 8 次，平均停留 4 分 32 秒]
[升级为数据室]  ← 主按钮
把补充材料打包给投资人，并开启 NDA 与访问审批。
```

### 三、决策层：把绝对表述改为带边界的规则

#### 6. “文件实体永远只落在文档库”需要加边界
这句话方向对，但过于绝对。未来可能出现以下合理例外：

- **Room 专属临时草稿**：用户上传了一个还未决定放入哪个文件夹的文件，是否需要先出现在 Library？
- **仅用于 Room 的 logo/NDA 模板**：这类文件是否也要进文档库？

建议改写为：

> **默认原则**：上传动作可以发生在任何场景，但文件实体首先进入文档库。特殊文件类型（如 Room 封面、NDA 模板）可单独定义是否纳入文档库，但必须在数据模型中显式标记，避免成为“隐形资产”。

#### 7. 指标目标需要说明来源或校准机制
文档中的目标值（如 Link → Room 转化率 ≥ 15%）看起来是拍脑袋的。建议：

- 如果是引用 PRD 中的假设 H-04（数据室创建用户升级率 ≥ 15%），应明确标注来源。
- 如果是新设指标，应说明“上线后第 1 周根据 baseline 校准”。

建议新增：

| 指标 | 初始目标 | 来源 / 校准方式 |
|------|----------|-----------------|
| 单文档 Link → 数据室转化率 | ≥ 15% | 引用 PRD H-04；上线 14 天后根据实际 baseline 调整 |
| 数据室中选择已有文件比例 | ≥ 60% | 基于“统一资产模型”假设；若低于 40% 需重新审视上传路径 |
| 数据室创建成功率 | ≥ 80% | 可用性测试目标 |

#### 8. 增加「设计假设与验证方法」表
每个核心设计决策都应对应一个可证伪的假设：

| 设计决策 | 设计假设 | 验证方法 | 失败信号 |
|----------|----------|----------|----------|
| 所有文件先入库 | 用户能接受“数据室里的文件也在文档库” | 观察 Room 中选择已有文件的比例 | < 40% |
| 权限不互相覆盖 | 用户能区分 Link 和 Room 两种共享路径 | 可用性测试 + 客诉率 | 权限相关客诉 > 5% |
| 默认跟随最新版本 | 用户希望 Room 中的材料保持最新 | Room 中版本锁定功能使用率 | 锁定率 > 50% 说明默认策略需调整 |
| “升级为数据室”按钮 | 单文档热度高时用户愿意升级 | Link → Room 转化率 | < 5% |

### 四、落地层：增加可执行性

#### 9. 增加「不做清单（Non-Goals）」
明确本设计不解决什么，避免范围蔓延：

- 文档库本期不支持内部文件夹（未来可评估）。
- 数据室本期不支持文档的协同编辑或评论（Q&A 是独立模块）。
- 不支持 Room 内文档“仅 Room 可见”的隐藏模式（违反统一资产原则）。
- 不支持数据室作为独立 SKU 对外售卖（保持统一平台逻辑）。

#### 10. 增加「下一步与责任人」
产品形态文档不应该只定义“是什么”，还要定义“怎么推进”：

| 事项 | 责任人 | 交付物 | 截止时间 |
|------|--------|--------|----------|
| 确认当前 Room 上传是否统一写 documents 表 | 后端负责人 | 代码走查报告 | T+2 |
| 设计文档详情页“暴露面”组件 | 设计负责人 | UI 稿 + 文案 | T+5 |
| 制定权限冲突决策矩阵的测试用例 | QA 负责人 | 测试用例集 | T+7 |
| 5-8 人可用性测试 | 产品负责人 | 测试报告 | T+14 |

3. **落地层**：告诉团队“下一步谁做什么”（gap 分析、责任人、验证计划）。
