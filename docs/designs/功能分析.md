**结论先行：不要“完全独立”，也不要“混为一谈”。最佳设计是「底层一份文档资产，上层两种场景表达」。**  
文档库（Documents）是唯一的素材与解析源头；数据室（Deal Room）是在文档库之上建立的“项目级共享容器”，带文件夹、成员、NDA、审批、Q&A。上传是一个无处不在的能力，但上传后的产物必须统一归入文档库。

## 一、从用户 JTBD 看心智模型

DealSignal 的核心用户有三类，他们对“上传文档”和“数据室”的期待并不相同：

| 角色 | 典型任务 | 心理账户 |
|------|----------|----------|
| **融资创始人** | “把 Pitch Deck 发给投资人，看谁看了、看了多久、对哪页感兴趣” | **单文档快速分享**，越轻越好 |
| **投资机构 IR** | “给 LP 发季度报告，或把尽调资料打包给多个投资人” | **批量、可控、合规分发**，需要容器 |
| **B2B 销售 AE** | “发 Proposal 给客户，识别购买信号；复杂项目再补资料包” | 先**单文档**，成交周期拉长后升级**资料包** |

关键洞察：

- **单文档分享**和**数据室**是两种不同的“Job”，但共享同一种“材料”——PDF / Office 文档。
- 用户不会说“我要上传一个文档到数据室”，他们会说“我要把尽调资料整理成一个 Room”。
- 如果强制所有上传都必须先创建数据室， Pitch Deck 这种高频轻量场景会被拖重，激活率下降。
- 如果完全独立，用户会在两个池子里重复上传同一文件，AI、版本、分析全部割裂。

## 二、三种方案的产品级对比

| 维度 | A. 完全独立（两个文档池） | B. 底层统一 + 上层场景化（推荐） | C. 全部围绕数据室上传 |
|------|---------------------------|----------------------------------|------------------------|
| **心智模型** | 清晰：文档=文件，数据室=项目 | 需要一点学习，但符合“素材→作品”逻辑 | 简单一致，但违背轻量分享习惯 |
| **激活摩擦** | 中（先选模块） | 低（默认单文档，需要时再组 Room） | 高（每次都要建 Room） |
| **文档复用** | 差，同一文件在两个池子各存一份 | 优，Room 只是文档的引用/集合 | 中，文件被 Room 绑定 |
| **AI / 搜索 / 分析** | 需重复建设，数据孤岛 | 统一解析、统一 embedding、统一联系人画像 | 统一，但scope以Room为单位 |
| **版本控制** | 极易分叉 | 文档库是版本源头，Room 可选固定/跟随最新 | 版本被 Room 隔离 |
| **权限复杂度** | 简单（各自为政） | 中等（需处理文档链接 vs Room 权限的叠加） | 简单 |
| **工程成本** | 高（两套解析、存储、Viewer） | 中（一套资产模型 + 集合层） | 中 |
| **商业化路径** | 难做 upsell | 自然：单文档免费/低价 → Room 团队版 | Room 一上来就是付费门槛 |

**结论：B 是唯一兼顾用户效率、技术可持续和商业扩展的方案。**

## 三、推荐方案的产品形态

### 3.1 一句话架构
> **“文档库是仓库，数据室是展厅；所有展品必须先从仓库出库，但展厅可以决定谁来、怎么看、要不要签 NDA。”**

### 3.2 信息架构建议

保留当前 `Dashboard / Documents / Links / Deal Rooms / Contacts / Insights / Settings` 的顶层导航，但明确各自定位：

- **Documents（文档库）**：所有原始素材的家。上传、解析、版本、AI 问答、单文档分析都在这里。
- **Links（链接）**：文档的“可追踪分享实例”。一个文档可以生成多个链接，对应不同接收方/权限。
- **Deal Rooms（数据室）**：面向特定交易/项目/人群的“多文档容器”。它引用文档库中的文件，并叠加文件夹结构、成员、审批、NDA、Q&A。

### 3.3 上传入口：场景化但归宿统一

| 入口 | 用户意图 | 行为 |
|------|----------|------|
| 全局“上传文档” | 先存文件，再决定怎么用 | 进入文档库 |
| 文档库页“上传” | 归档/管理文件 | 进入文档库 |
| 智能链接创建器“选择/上传文档” | 分享 | 上传后进入文档库，并生成 Link |
| 数据室详情页“上传文件” | 补充资料 | 上传后进入文档库，**同时自动加入当前 Room 的当前文件夹** |
| 数据室创建向导“选择/上传文档” | 批量建 Room | 文件先进文档库，再被选中加入 Room |

**原则：上传动作可以发生在任何场景，但文件实体永远只落在文档库。**

## 四、必须做出的 7 个关键产品决策

### 1. 文档是原子资产，Room 是引用关系
- 数据室不“拥有”文件，只“引用”文件（`deal_room_documents` 这类 join table 是正确的）。
- 同一份 Pitch Deck 可以同时出现在“文档库链接”和“多个数据室”里。

### 2. 版本策略：默认跟随最新，但 Room 可锁定版本
- 文档库更新文件后，Room 默认展示最新版。
- Room  owner 可以显式“锁定当前版本”，避免尽调过程中材料被替换引发争议。
- 旧版本仍保留，便于审计。

### 3. 权限模型：链接权限 与 Room 权限 并行，但需透明
- 文档级 Link：控制谁可以通过这个链接看。
- Room 级权限：控制谁能进入 Room，以及能看到哪些文件夹。
- **不要互相覆盖**，但要在 UI 上明确告知暴露面：

> “此文档已通过 3 个链接 + 2 个数据室 共享。”

### 4. 删除/归档策略
- 文档库删除文件时，如果该文件被任何 Room 引用，**禁止硬删除**，提供两种选择：
  - **归档**（文件从 Library 隐藏，但 Room 中仍可见）；
  - **从所有 Room 移除后再删除**。
- 推荐默认“归档”，保证已发出去的 Room 不会突然缺页。

### 5. AI 与搜索的天然优势
- 因为底层统一，AI 问答可以灵活切换 scope：
  - 当前文档
  - 当前数据室
  - 整个工作区文档库
- 这是独立方案做不到的体验，应作为核心卖点。

### 6. 联系人/热度评分跨场景聚合
- 一个投资人既可能通过 Link 看了你的 Deck，也可能在 Room 里看了 Financial Model。
- 热度评分必须合并所有触点，而不是 Link 一套、Room 一套。

### 7. 商业化包装
- **单文档分享 + Link**：免费/个人计划，拉新。
- **数据室 + 成员/审批/NDA/Q&A**：团队/企业计划， upsell。
- 产品内设计“升级为数据室”的入口（例如投资人看完 Deck 后创始人一键把补充材料打包成 Room），是最高效的付费转化路径。

## 五、需要警惕的陷阱

| 陷阱 | 后果 | 缓解方式 |
|------|------|----------|
| **Room 里的上传文件不在文档库显示** | 用户找不到源文件，AI 搜索遗漏 | 强制所有文件入库，Room 只是视图 |
| **文档删除后 Room 里变成死链** | 尽调资料突然缺失，信任崩塌 | 默认归档，删除前校验引用 |
| **Link 权限与 Room 权限互相矛盾** | 用户以为撤了 Room 就安全了，其实 Link 还在 | 在文档详情页统一展示“暴露面” |
| **Room 被当成文件夹滥用** | 内部文件管理也建 Room，导致权限混乱 | 文档库后续可补文件夹功能，Room 定位为“对外共享容器” |
| **数据室创建步骤太重** | 创始人只想发 Deck，却被迫建 Room | 保留单文档 Link 的快速路径 |

---

## 六、如果要推翻当前设计，我建议这样改

1. **删除“数据室独立上传”的伪概念**：Room 里的上传 API 本质上还是 `POST /documents`，然后自动 `POST /deal-rooms/{id}/documents`。
2. **文档库加一列“已用于”**：显示每个文档被哪些 Link / Room 引用，帮助用户管理暴露面。
3. **Room 详情页文件区 = 文档库子集 + 文件夹结构**：支持从库中拖拽/选择已有文档，而不是只能上传新文件。
4. **Dashboard 信号流合并两种场景**：投资人看了 Link 里的 Deck，又在 Room 里看了财务模型，应该合并为同一个 Hot Signal。
5. **上线后重点验证的指标**：
   - 单文档 Link → 数据室的转化率
   - 数据室中“从文档库选择已有文件”的比例（越高说明统一模型越成功）
   - 同一文件在 Link + Room 中的复用率
   - 用户是否能正确理解“文档库”和“数据室”的关系（可用性测试）

---

### 最终一句话

**DealSignal 的“上传文档”和“数据室”应当是「同一资产在不同场景下的两种表现」：文档库是唯一的素材与真相来源，数据室是面向交易场景的高级共享容器。完全独立会制造数据孤岛，完全合并会拖重核心分享场景；底层统一、上层解耦，才是产品、技术与商业的最优解。**
