> **会议目标**：对 `docs/PRD-v2.1.0.md` 进行结构化评审，确认需求基线、范围边界、验收标准、技术可行性、设计可行性与合规性，决定是否批准进入 TDD 阶段。
> **文档状态**：已归档  
> **建议时长**：2.5 ~ 3 小时（可分两场：上半场产品/设计/测试，下半场技术/运营/合规）

## 3. 评审议程与待评审项

### 议程一：文档基线与范围（20 分钟）

1. 确认 PRD 版本号统一为 `v2.1.0`，旧 PRD 已归档为 `docs/PRD-v2.0.7-ARCHIVED.md`。
2. 确认 front matter 中的 `linked_docs`、`open_questions`、`ai_red_flags`、`pending_confirmation` 是否完整。
3. 评审第 4 节范围与边界：
   - In Scope 的 9 项是否全部合理？
   - Out of Scope 的 9 项是否明确且无异议？
   - 依赖项与风险等级是否被接受？
4. 评审成功标准（3.4 节）：指标、基线、目标值、观察周期是否可衡量。

### 议程二：用户与关键路径（20 分钟）

1. 评审第 5.1 用户画像与 5.2 JTBD：三类用户是否覆盖核心场景？
2. 评审第 5.3 用户旅程地图：认知 → 激活 → 使用 → 留存 → 推荐是否完整？
3. 评审第 5.4 关键用户路径与 AC 映射：
   - 路径 A：AC-01 ~ AC-08
   - 路径 B：AC-09 ~ AC-12
   - 路径 C：AC-13 ~ AC-17, AC-32
   - 路径 D：AC-31
4. 关键路径是否覆盖了本次 MVP 的“上传 → 解析 → 分享 → 阅读 → 问答 → 洞察 → 跟进”闭环？

### 议程三：产品原则与设计约束（15 分钟）

1. 评审第 6.1 产品原则：6 条原则是否存在冲突？
2. 评审第 6.2 设计约束：响应式、状态、二次确认等是否可落地？
3. 评审第 6.3 技术约束：
   - “子域名 + Workspace”混合隔离模式是否可行？
   - Cloudflare URL Signing + 后端签名 URL 方案是否通过安全评审？
   - Go channel 异步队列是否满足 MVP 事件量？
4. 评审第 6.4 合规约束：GDPR / CCPA / 数据安全法要求是否足够？

### 议程四：功能需求逐条评审（40 分钟）

按模块逐条确认第 8 节 16 个 FR：

| 模块 | FR 编号 | 评审重点 |
|------|---------|----------|
| 上传与解析 | FR-01 ~ FR-02 | 文件大小、去重、断点续传、Office 转换依赖 |
| 查看与渲染 | FR-03 ~ FR-04 | 签名 URL 有效期、Canvas 渲染、行为上报精度 |
| AI 助手 | FR-05 ~ FR-06 | hybrid search、evidence 引用、高亮坐标换算 |
| 链接与权限 | FR-07 ~ FR-09 | 权限策略组合、密码安全、水印防篡改 |
| 意图分析 | FR-10 ~ FR-11 | 热度评分权重、提醒合并策略、跟进建议生成 |
| 数据室 | FR-12 ~ FR-13 | NDA gating、folder 级权限、审批流程 |
| 通知 | FR-14 | 邮件合并、退订、硬退回处理 |
| 集成 | FR-15 ~ FR-16 | HubSpot/Salesforce 字段映射、Slack 推送规则 |

每条 FR 需确认：
- 需求描述是否无歧义
- 前置/后置条件是否清晰
- 业务规则是否可执行
- 异常处理是否完整
- 关联 AC / EVT / API 是否正确

### 议程五：数据架构与接口契约（30 分钟）

1. 评审第 10.1 数据模型：
   - 所有表是否包含 `tenant_id` + `workspace_id`？
   - `room_members` 以 email 作为准入标识的设计是否可接受？
   - `chunk_boxes` 的 `PAGE_IMAGE_NORMALIZED` 坐标方案是否可行？
2. 评审第 10.2 接口契约：
   - 内部 API 基础路径 `/{workspaceSlug}/api/v1` 是否被接受？
   - API-09 公开链接使用自定义域名 + query 参数是否最终确定？
   - API-14 `/public/deal-rooms/{roomId}/access-requests` 是否满足公开访问需求？
3. 确认是否需要在 PRD 中补充 OpenAPI 详细契约，或放到 TDD/API-SPEC 中。

### 议程六：非功能需求与验收标准（25 分钟）

1. 评审第 9 节 NFR：
   - 性能指标（上传 P99 ≤ 2s、AI 问答 P95 ≤ 3s 等）是否可达成？
   - 安全要求（签名 URL、租户隔离、审计日志）是否完整？
   - 可用性目标 ≥ 99.5% 是否合理？
   - 新增 9.7 Accessibility 与 9.8 数据保留是否满足合规要求？
2. 评审第 12 节 32 条 AC：
   - 是否每条 AC 都可测试、可验证？
   - 正常路径 / 异常路径 / 边界 / 权限 / 性能 / 安全是否覆盖？
   - 12.3 验收检查清单是否逐项确认并勾选？

### 议程七：交互与体验（20 分钟）

1. 评审第 11.2 信息架构：Dashboard / Documents / Links / Deal Rooms / Contacts / Insights / Settings 是否完整？
2. 评审第 11.3 关键页面清单：9 个核心页面设计稿排期（2026-07-09）是否可行？
3. 评审第 11.4 交互说明与 11.5 页面状态规范：加载态、空态、错误态、权限不足态是否覆盖？
4. 评审第 11.6 文案规范：关键场景文案是否一致、无歧义？

### 议程八：集成、实施计划与风险（20 分钟）

1. 评审第 13 节集成与接口：
   - 第三方依赖（OnlyOffice、OpenAI、Cloudflare、SendGrid/SES 等）是否确认？
   - 备选方案是否可接受？
2. 评审第 14 节实施计划：
   - 23 个 TASK 的工时与依赖是否合理？
   - 里程碑日期（PRD 评审 06-25、TDD 确认 07-02 等）是否可行？
3. 评审第 15 节风险与缓解：
   - R-01 文档解析失败率、R-02 AI 幻觉、R-05 权限泄露等高风险项是否已制定应急预案？

### 议程九：上线、运维与合规（15 分钟）

1. 评审第 16 节上线与运维：
   - 内测 → 灰度 → 全量策略是否合理？
   - 回滚条件与步骤是否明确？
   - On-Call 安排是否到位？
2. 评审第 17 节合规、安全与隐私：
   - 隐私政策、DPA、数据导出/删除流程是否能在上线前完成？
   - AI 问答日志不用于模型训练是否可承诺？

### 议程十：决策记录与待确认问题（15 分钟）

1. 评审第 18 节决策记录 D-01 ~ D-25：
   - 是否有需要推翻或重新讨论的决策？
   - D-25（冻结 v2.0.7 并重构为 v2.1.0）是否被接受？
2. 清空或转化第 19.6 节待确认问题清单：
   - 异步任务队列最终选型
   - LLM / Embedding 服务商与模型版本
   - Cloudflare URL Signing 密钥轮转
   - Viewer 动态水印元数据签名
   - 设计稿产出排期
   - TDD / ARCHITECTURE / database-model 编写计划
   - 隐私政策与 DPA 确认
