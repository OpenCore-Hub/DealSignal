**评审原则**：
1. **产品优先**：只做能直接提升用户成交效率或安全可信度的事。
2. **技术务实**：不做为了架构而架构的改造；优先消灭数据质量与安全隐患。
3. **敢于剔除**：对 ROI 低、依赖重、代码已烂尾的任务直接说“不”。
4. **敢于合并**：把重复或弱耦合的任务并成一个有明确验收标准的任务。
5. **不拍马屁**：优点只说确实能形成壁垒的，缺点直接点名。

## 2. 设计层面：哪些是优秀设计，哪些是过度设计

### 2.1 必须保留的优秀设计

| 设计点 | 为什么优秀 |
|---|---|
| **Link 作为分享唯一实体** | 避免了 Deal Room 与文档链接两套并行体系，长期维护成本低。 |
| **Share / Invite / Access 三入口** | 用户意图清晰：分发、识别、授权。比把所有开关堆在一个 sheet 里好得多。 |
| **Access Rules fail-closed** | block 优先、allow 必须命中，符合数据室“宁可错杀”的安全心智。 |
| **邀请自动加入 allow list** | 产品闭环，减少用户手动同步两份列表的认知负担。 |
| **规则变更使旧 session 失效** | 满足“修改后立即生效”的预期，避免已签发 session 成为安全后门。 |
| **公共 AI 按 link+visitor 隔离** | 既防止跨链接泄露，又为后续意图分析留了数据入口。 |

### 2.2 应当警惕的过度设计

| 设计点 | 问题 | 建议 |
|---|---|---|
| **Preset Custom 状态 + 二次确认** | 对 MVP 来说过于精细。用户绝大多数时候只选 public/standard/confidential，custom 状态的使用频率不足以支撑一套复杂的偏离检测。 | 简化：预设切换时直接覆盖，给出 Snackbar 提示即可；custom 标签可以保留，但不需要二次确认。 |
| **Access Tab 15 个开关** |  Huntress 设计里 Advanced 区域有 AI Agents、file requests、index file、QA conversations 等开关。当前产品阶段这些能力要么不存在，要么没人用，一次性暴露会吓跑用户。 | 只保留已实现的开关；未实现的能力不要放 UI 占位。 |
| **服务端动态图片水印** | 对每页图片做服务端水印需要渲染管道改造，成本高、延迟大。对当前目标客户来说，前端水印 + 防截图已足够。 | 先做“后端生成水印文案 + 前端渲染 + 防绕过”，服务端图片水印放到企业版 backlog。 |
| **NDA 版本化 + 电子签名** | 真实法律场景确实需要，但当前客户规模下 NDA 只是勾选框。 | 保留简单 NDA 勾选，版本化放到 P2 后期。 |
| **工作区级分享策略** | 长期来看必要，但当前管理员数量和使用频次不足以支撑。 | 延后到中期末，先让单 Link 规则跑稳。 |

## 3. 技术层面：哪些是硬核缺陷，哪些是无关紧要的洁癖

### 3.1 真正的 P0 缺陷（不做会埋雷）

| 缺陷 | 严重程度 | 为什么严重 |
|---|---|---|
| **去重先标记 Redis 再写 DB** | 🔴 P0 | DB 写入失败（如 max_access 达到、连接断开）会导致 Redis key 仍在，后续真实访问被永久吞掉。这是数据质量事故。 |
| **邀请 token 明文存储** | 🔴 P0 | DB 泄露即可伪造邀请链接。任何有 DB 只读权限的人都能冒充受邀者。 |
| **通知邮件同步发送** | 🔴 P0 | `notification.Service.Enqueue` 直接发邮件，SMTP 超时或限流会拖慢 API，且无重试。 |
| **签名 URL 缺失** | 🟡 P1 | 页面图片和下载 URL 可被转发盗用。虽然还没被攻击，但这是企业客户采购时的必查项。 |

### 3.2 可以容忍的短期妥协

| 妥协点 | 为什么现在可以忍 |
|---|---|
| **Access Rules 全量替换** | 简单、幂等、够用。先补审计快照即可，不必急着做行级 CRUD。 |
| **Session 失效用 `updated_at`** | 实现简单，正确性足够。未来再引入 `security_version`。 |
| **Key Page 用文档标题粗匹配** | 比停留 3 秒更合理，且实现简单。先跑起来，再按页细化。 |
| **水印用前端 CSS 覆盖** | 虽然可绕过，但对当前 SMB 客户够用。先补后端文案和防截图，再考虑服务端水印。 |

### 3.3 纯粹是垃圾的技术债

| 垃圾项 | 说明 | 处理 |
|---|---|---|
| `SMTP_USER` 作为通知收件人兜底 | 通知本应发给创建者，环境变量兜底会把测试/系统邮箱当成收件人，导致产品不可用。 | 立即移除。 |
| `CreateLinkSheet` 仍作为独立入口 | 当前 `DealRoomShareDialog` 已经覆盖新建/编辑，`CreateLinkSheet` 是 mock 且与三 Tab 模型冲突。 | 下线或重写为“新建模式”嵌入 `DealRoomShareDialog`。 |
| 前端自行构造水印文本 | 后端不控制内容，无法加入 IP 等安全信息，且不同页面可能不一致。 | 后端统一生成。 |

## 5. 剔除与延后的逻辑

### 5.1 为什么剔除 LONG-005 预测性 Lead Scoring

- **数据不足**：DealSignal 当前客户量级不足以支撑可泛化的预测模型。
- **标签未定义**：“转化”是回复邮件、进入谈判、还是签约？产品自己都没想清楚。
- **可解释性风险**：企业销售需要知道“为什么这个 contact 是 85 分”，黑盒模型会导致销售不信任。
- **替代方案**：把 heat score + 关键事件（访问关键页、下载、转发、AI 高意向问题）做成规则版评分，足够支撑前 100 个付费客户。

**结论**：LONG-005 不是“长期”，是“伪需求”。剔除，等数据量超过 10k 转化事件后再评估。

### 5.2 为什么延后 LONG-004 CRM 深度集成

- 高价值，但重。
- 当前信号体系还没跑通，直接往 CRM 写 timeline 会写出大量垃圾活动。
- 建议先完成 MID-003 通知规则引擎 + MID-007 Link Analytics，让“什么值得同步到 CRM”有清晰标准，再集成。

### 5.3 为什么合并 MID-004 与 MID-006

- MID-004 的“Canvas 动态水印”前端部分已基本完成。
- MID-006 的“服务端水印与防绕过”本质上是在 MID-004 基础上补后端文案和安全措施。
- 两个任务并行会导致 UI 与后端不同步。合并为一个“可信水印”任务，验收标准更清晰。

## 7. 产品设计上的取舍建议

### 7.1 砍掉未实现能力的占位开关

当前 Access Tab 规划里有 AI Agents、file requests、index file generation、QA conversations 等开关。这些能力不存在或无人使用，放在 UI 里只会：
- 增加认知负荷
- 让用户怀疑产品是否完整
- 给后端留下无意义的字段

**建议**：Access Tab 只保留已实现的 7~8 个开关：
- Require email / OTP / password
- Allowed / Blocked viewers
- Watermark
- NDA
- Allow downloading
- AI Copilot（如果公共 AI 已启用）

其余 Advanced 开关全部下线，等能力实现后再放出来。

### 7.2 简化 Preset 行为

不要搞“偏离预设 → custom → 二次确认覆盖”的复杂状态机。

**建议**：
- 用户选 preset：直接覆盖 Access Tab 开关，显示 Snackbar “Standard preset applied”。
- 用户手动修改开关：preset 下拉显示 “Custom”。
- 用户再次选 preset：直接覆盖，不二次确认。

这是 90% 场景下的最优解，省下 30% 的前端状态管理复杂度。

### 7.3 把 Analytics Tab 提前到 P1

数据室用户付费的核心动机之一是“知道谁在认真看我的材料”。Link Analytics 比 AI 意图分析、CRM 集成更能直接证明产品价值。

**建议**：MID-007 与 MID-003 并行，甚至优先于 LONG 任务。

### 7.4 访问请求（Request Access）不要做得太重

被 block 的访客点击 request access，提交 email + reason，创建者收到邮件通知，一键 approve。

**不要做**：审批工作流、多级审批、自动拒绝规则、访客身份验证。这些对 MVP 是过度设计。

## 8. 技术架构上的取舍建议

### 8.1 去重：先写 DB，再标记 Redis

这是 Wave 1 必须修复的 P0。两种方案：

**方案 A（推荐）**：
```go
rows, err := s.queries.RecordLinkOpened(...)
if err != nil { return err }
_ = s.dedup.MarkOpen(...) // DB 成功后再标记
```

**方案 B（更严谨）**：
- 生成幂等 ID（如 `linkID:visitorID:timestampHour`）。
- DB insert 使用 `ON CONFLICT DO NOTHING`。
- Redis 仅作为缓存，不决定写入。

方案 A 改动最小，先落地。

### 8.2 邀请 token：立即 hash

```go
token := generateRandomToken()
hash := sha256.Sum256([]byte(token))
// store hash in DB
// return original token to client only once
```

解析时按 hash 查询。历史 token 可一次性迁移：读出 → hash → 更新列 → 删除明文。

### 8.3 通知：全部异步

`notification.Service.Enqueue` 应该把所有邮件（包括邀请、访问通知、验证码）统一写入 `notifications` 表，由 worker 消费。

不要把“email 立即发送”和“slack 入队”混在一起。

### 8.4 水印：后端生成文案，前端渲染 + 防绕过

不要碰服务端图片水印。对当前阶段来说：
- 后端生成 `watermarkText`。
- 前端 Canvas 上方叠加。
- 监听 `keydown` Print Screen、右键、`beforeprint`。
- 后端记录安全事件 `screenshot_attempted`（可选）。

这是成本收益比最高的方案。

### 8.5 签名 URL：与 LinkSession 解耦

签名 URL 有效期建议 15 分钟，独立于 15 分钟滑动的 LinkSession。这样：
- 页面图片可以缓存 15 分钟。
- session 失效后图片 URL 也自然失效。
- 不要搞“签名 URL 与 session 绑定”的复杂逻辑。
