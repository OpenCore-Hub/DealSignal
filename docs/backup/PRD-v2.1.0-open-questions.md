## 一、技术架构类（P0）

### Q-T01：异步任务队列最终选型
- **问题描述**：MVP 使用 Go channel / 本地队列，但目标上线后 90 天事件量可能增长。是否满足未来扩展？
- **当前假设**：MVP 使用 Go channel；日事件量达到千万级后二期引入 RabbitMQ / SQS / Kafka。
- **需要确认**：
  - Go channel 在当前预期并发下的可靠性
  - 死信队列、重试、堆积监控如何实现
  - 二期迁移到消息中间件的成本
- **决策人**：技术负责人
- **建议截止日期**：2026-07-02（TDD 确认前）
- **推荐结论**：接受 Go channel + 明确监控/降级方案，或在 TDD 中给出切换阈值。

### Q-T02：LLM / Embedding 服务商与模型版本
- **问题描述**：AI 问答与 Embedding 使用哪家服务？是否需支持多供应商降级？
- **当前假设**：OpenAI-compatible API；备选 Azure OpenAI / 自托管 bge + vLLM。
- **需要确认**：
  - 主供应商与模型版本（如 gpt-4o / text-embedding-3-large）
  - 调用方式（API / SDK / 自托管）
  - 数据出境合规性
  - 成本估算与限流策略
- **决策人**：技术负责人 + 合规负责人
- **建议截止日期**：2026-07-02
- **推荐结论**：MVP 使用 OpenAI API，预留 Azure OpenAI 降级开关。

### Q-T03：Cloudflare URL Signing 密钥管理
- **问题描述**：签名 URL 由后端签发、CDN 边缘校验，密钥如何生成、存储、轮转？
- **当前假设**：后端统一签发，CDN 边缘校验；密钥按季度轮转。
- **需要确认**：
  - 密钥存储位置（KMS / Secrets Manager）
  - 签发算法与字段（tenantId / workspaceId / documentId / pageNumber / purpose / expiresAt）
  - 密钥轮转时的平滑过渡策略
  - 签名 URL 不携带 userId 是否影响审计追踪
- **决策人**：技术负责人 + 安全负责人
- **建议截止日期**：2026-07-02
- **推荐结论**：使用 AWS Secrets Manager / HashiCorp Vault 存储，轮转时保留新旧密钥双活 24h。

### Q-T04：Viewer 动态水印元数据签名
- **问题描述**：水印信息（邮箱、时间、IP）由后端签名后下发，前端 Canvas 绘制。签名字段与防篡改策略是什么？
- **当前假设**：水印元信息来自后端签权，前端通过 Canvas 绘制；截图时水印仍保留。
- **需要确认**：
  - 水印字段具体包含哪些（邮箱、访问时间、IP、visitor_id？）
  - 签名 token 有效期多长
  - 前端水印被绕过时的兜底方案（如审计日志）
  - 是否支持“截图追溯”功能
- **决策人**：技术负责人 + 产品负责人
- **建议截止日期**：2026-07-09（设计稿确认前）
- **推荐结论**：水印字段 = 邮箱 + 访问时间 + IP 哈希；token 与签名 URL 同生命周期；审计日志兜底。

### Q-T05：公开链接自定义域名与 API 路径样式
- **问题描述**：公开链接使用品牌方自定义域名 + query 参数（`?tenant=&workspace=&token=`）。是否改为路径形式更优雅？
- **当前假设**：自定义域名根路径 + query 参数传递 tenant/workspace/token。
- **需要确认**：
  - query 参数是否会被浏览器/代理缓存/日志泄露
  - 是否改为 `/{token}` 路径形式
  - 自定义域名 CNAME 验证与 SSL 自动签发流程
- **决策人**：技术负责人 + 产品负责人
- **建议截止日期**：2026-07-02
- **推荐结论**：MVP 保留 query 参数形式，但 token 需足够随机且短期有效；TDD 中评估路径形式兼容性。

### Q-T06：Office 转换资源规格与部署方式
- **问题描述**：OnlyOffice 自托管独立集群（4C8G × 2）是否满足并发与稳定性？
- **当前假设**：独立 K8s 集群，并发转换 10 个文档，VPC 对等连接。
- **需要确认**：
  - 实际压测数据
  - 失败重试与降级到 LibreOffice 的触发条件
  - 转换缓存清理策略
- **决策人**：技术负责人
- **建议截止日期**：2026-07-23（基础服务完成前）
- **推荐结论**：按当前规格部署，内测期收集转换成功率数据，< 95% 时触发降级预案。

## 二、设计与体验类（P1）

### Q-D01：核心页面设计稿产出排期
- **问题描述**：Dashboard、Viewer、AI 对话框、数据室等 9 个核心页面设计稿尚未产出。
- **当前假设**：计划在 2026-07-09 前完成高保真设计稿与交互原型。
- **需要确认**：
  - 设计资源是否足够
  - 是否需要分优先级（P0 页面优先）
  - 设计稿评审流程
- **决策人**：设计负责人
- **建议截止日期**：评审会上确认
- **推荐结论**：Viewer / 上传页 / Dashboard 优先；数据室/设置页可稍后。

### Q-D02：AI 助手默认展开还是收起
- **问题描述**：PRD 要求 AI 助手默认收起，但是否在首次查看时显示引导提示？
- **当前假设**：默认收起，不遮挡文档内容。
- **需要确认**：
  - 首次使用是否显示功能引导
  - 未登录访客是否能看到 AI 助手
  - AI 助手在移动端的交互方式
- **决策人**：设计负责人 + 产品负责人
- **建议截止日期**：2026-07-09
- **推荐结论**：默认收起，首次查看显示一次性引导气泡；访客可用 AI 但限制提问次数。

### Q-D03：权限强度滑块的具体档位
- **问题描述**：“权限强度滑块”在交互说明中提及，但具体档位未定义。
- **当前假设**：实时显示安全强度与接收方摩擦。
- **需要确认**：
  - 档位划分（如公开 / 邮箱验证 / 白名单 / 密码 / NDA）
  - 每个档位对应的默认配置
  - 移动端是否显示摩擦提示
- **决策人**：产品负责人 + 设计负责人
- **建议截止日期**：2026-07-09
- **推荐结论**：三档：低摩擦（公开/邮箱验证）、中强度（白名单/密码）、高强度（NDA/白名单+密码）。

## 三、产品范围类（P1）

### Q-P01：热度评分算法是否需更详细定义
- **问题描述**：FR-10 使用规则加权评分，但具体权重、关键页定义、Hot/Warm/Cold 阈值未量化。
- **当前假设**：规则可解释，二期引入 ML。
- **需要确认**：
  - 三类角色的权重差异
  - Hot（≥70）、Warm（40-69）、Cold（<40）阈值是否合适
  - 是否需要 A/B 测试验证
- **决策人**：产品负责人 + 数据负责人
- **建议截止日期**：2026-07-09
- **推荐结论**：MVP 使用规则加权并公开因子；上线后 14 天根据数据校准阈值。

### Q-P02：数据室模板默认结构
- **问题描述**：FR-12 提到 Seed / Series A / LP Update / Sales Proposal 模板，但默认文件夹结构未定义。
- **当前假设**：创建数据室时自动生成标准文件夹结构。
- **需要确认**：
  - 每种模板的具体文件夹列表
  - 是否允许用户自定义
  - 模板与权限的默认映射
- **决策人**：产品负责人
- **建议截止日期**：2026-07-09
- **推荐结论**：每种模板提供 5~8 个默认文件夹，用户可增删改。

### Q-P03：存储配额与超限行为
- **问题描述**：FR-01 提到“租户存储配额未超限”，但配额值与超限行为未定义。
- **当前假设**：按 plan 区分配额（free / founder / pro / secure_room）。
- **需要确认**：
  - 各 plan 的存储上限
  - 超限后是否禁止上传或仅警告
  - 是否支持购买扩容
- **决策人**：产品负责人
- **建议截止日期**：2026-07-02
- **推荐结论**：MVP 仅做软限制与提示，付费扩容放到二期。

## 四、合规与安全类（P1）

### Q-C01：隐私政策与数据处理协议（DPA）
- **问题描述**：PRD 要求法务确认隐私政策与 DPA，但完成时间未明确。
- **当前假设**：可在上线前完成。
- **需要确认**：
  - 隐私政策起草责任人
  - DPA 是否面向企业客户
  - 用户数据导出/删除流程的 UI 入口
- **决策人**：合规负责人
- **建议截止日期**：2026-09-10（测试通过前）
- **推荐结论**：隐私政策必须在灰度前上线；DPA 可先行提供模板。

### Q-C02：接收方数据追踪的知情同意
- **问题描述**：PRD 要求 Viewer 页面展示追踪说明，但具体文案与交互未定义。
- **当前假设**：明确告知接收方数据追踪范围，并提供隐私政策入口。
- **需要确认**：
  - 是否需要首次访问时弹窗确认
  - 是否允许接收方 opt-out（部分追踪）
  - 不同地区（GDPR/CCPA/中国）的文案差异
- **决策人**：合规负责人 + 产品负责人
- **建议截止日期**：2026-07-09
- **推荐结论**：首次访问显示底部横幅说明 + 隐私政策链接；暂不提供 opt-out，记录于隐私政策中。

### Q-C03：AI 问答内容的毒性/敏感过滤
- **问题描述**：PRD 要求对 LLM 输出做毒性/敏感内容过滤，但具体策略未定义。
- **当前假设**：后端统一过滤。
- **需要确认**：
  - 使用何种过滤机制（关键词 / 分类模型 / 供应商内置）
  - 过滤触发后的用户提示文案
  - 是否需要人工审核入口
- **决策人**：技术负责人 + 合规负责人
- **建议截止日期**：2026-07-09
- **推荐结论**：MVP 使用供应商内置 moderation + 敏感词兜底；用户可举报不当回答。

## 五、运营与增长类（P2）

### Q-O01：种子用户招募标准
- **问题描述**：上线策略中提到 20 个种子用户，但招募标准与反馈机制未定义。
- **当前假设**：内测期邀请 20 个种子用户。
- **需要确认**：
  - 种子用户画像（创始人 / IR / AE 比例）
  - 反馈收集方式（访谈 / 问卷 / 应用内反馈）
  - 是否提供 onboarding 1v1
- **决策人**：产品负责人 + 运营负责人
- **建议截止日期**：2026-09-10
- **推荐结论**：按 10 创始人 / 5 IR / 5 AE 招募；内测期每周 1 次访谈。

### Q-O02：邮件通知频率与合并策略
- **问题描述**：FR-11 提到同类事件合并，但合并窗口与频率未量化。
- **当前假设**：用户可配置提醒频率与事件类型。
- **需要确认**：
  - 默认合并窗口（如 5 分钟内同类事件合并）
  - 每日/每周摘要邮件是否提供
  - 退订后是否仍发送安全类通知
- **决策人**：产品负责人 + 运营负责人
- **建议截止日期**：2026-07-09
- **推荐结论**：默认 10 分钟合并；提供每日摘要开关；安全类通知不可退订。

## 六、建议评审会决策顺序

1. **先确认 P0 技术问题**（Q-T01 ~ Q-T05）：决定 TDD 能否启动。
2. **再确认产品范围问题**（Q-P01 ~ Q-P03）：决定范围是否需要调整。
3. **然后确认设计排期与交互问题**（Q-D01 ~ Q-D03）：决定设计里程碑是否可行。
4. **最后确认合规与运营问题**（Q-C01 ~ Q-C03, Q-O01 ~ Q-O02）：决定上线 checklist。
