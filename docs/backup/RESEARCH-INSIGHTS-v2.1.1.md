### 2.3 四大市场空白

#### 空白 1：从 Analytics 到 Action
> 多数工具告诉你"谁看了什么"，但不告诉你"现在该做什么"。

证据：
- DocSend 提供页面级分析，但跟进建议依赖用户手动判断
- Pitchwise 强调 engagement metrics，但 action assistant 不是核心
- Sizle 提供实时通知，但建议系统较浅

机会：构建"信号 → 解释 → 建议 → 一键执行"的完整闭环。

#### 空白 2：圈层专属语言
> 同一数据对创始人、IR、销售意味着完全不同的东西。

证据：
- 创始人关心"投资人热度"和"融资进度"
- IR 关心"LP engagement"和"合规审计"
- 销售关心"deal intent"和"pipeline"

机会：Dashboard 和 Insights 提供圈层视图切换，同一底层数据用不同语言呈现。

#### 空白 3：可解释 AI
> LLM 回答需要证据，证据需要定位到原文。

证据：
- 多数 AI 文档工具只提供聊天式回答
- Papermark 等开始探索 AI，但原文定位能力弱
- 投资人和客户对 AI 回答的信任度取决于可验证性

机会：AI 回答必须附带 evidence，并自动跳转到原文位置高亮。

#### 空白 4：轻量数据室
> 早期融资不需要 iDeals 级别的重 VDR，但需要比链接分享更结构化的空间。

证据：
- "Do I need a virtual data room for fundraising? Not initially."
- 尽调阶段才需要正式 data room
- 创始人希望"可升级"的分享方式

机会：提供场景模板化的轻量数据室，从 single link 平滑升级到 structured room。

## 3. 三层用户第一性原理需求

### 3.1 融资创始人

**第一性原理**：融资不是发材料，而是管理一群高价值决策者的注意力与信任。

**Top 6 需求**：

| 排名 | 需求 | 本质 | 当前替代方案痛点 |
|---|---|---|---|
| 1 | **识别真实兴趣** | 区分"礼貌打开"和"认真评估" | 邮件附件/网盘完全不可追踪 |
| 2 | **把握跟进时机** | 在兴趣最高时跟进，避免过早/过晚 | 只能靠感觉或统一群发 |
| 3 | **理解关注焦点** | 知道投资人在意团队/财务/市场 | 会议中才能获得反馈 |
| 4 | **控制材料安全** | 防止转发、下载、版本混乱 | 网盘权限粗粒度 |
| 5 | **缩短融资周期** | 减少尽调补资料反复拉扯 | 数据分散在邮件/网盘 |
| 6 | **维护专业形象** | 用品牌化 Viewer 替代邮件附件 | 显得不专业 |

**典型工作日**：
```
09:00  打开 Dashboard，看昨晚谁看了 deck
09:30  发现 Sarah 第 3 次查看财务页 → 系统建议跟进
10:00  一键生成 follow-up 邮件，补充财务模型
11:00  创建数据室，上传尽调资料
14:00  收到新投资人请求，发送智能链接
17:00  查看本周热度报告，调整下周路演策略
```

**决策关键词**：Investor Intent Score、热度、跟进、数据室、版本控制。

### 3.2 投资机构 / IR / LP Relations

**第一性原理**：IR 工作的核心是证明"我们在被看见"，并满足合规审计要求。

**Top 6 需求**：

| 排名 | 需求 | 本质 | 当前替代方案痛点 |
|---|---|---|---|
| 1 | **追踪 LP engagement** | 知道哪些 LP 看了季度报告 | 群发邮件无法追踪 |
| 2 | **满足合规审计** | 完整访问记录和审计日志 | 手动记录或缺失 |
| 3 | **精细化权限** | 不同 LP 看到不同材料 | 邮件附件无法控制 |
| 4 | **专业品牌门户** | 维护机构专业形象 | 普通网盘无品牌 |
| 5 | **管理 Q&A 与资料请求** | LP 重复问相同问题 | 邮件来回低效 |
| 6 | **数据主权与保留** | 合规保留与删除 | 担心数据泄露 |

**典型工作日**：
```
09:00  查看 LP Engagement Dashboard，发现 3 个 LP 未打开 Q4 report
10:00  发送提醒邮件给未打开 LP
11:00  处理 LP 在数据室 Q&A 模块的提问
14:00  导出审计日志给合规团队
15:00  准备下周 LP call 的 engagement 摘要
```

**决策关键词**：Engagement、审计、合规、白标、Q&A、保留策略。

### 3.3 B2B 销售 / AE / RevOps

**第一性原理**：销售的本质是识别购买意图并在正确时机集中火力。

**Top 6 需求**：

| 排名 | 需求 | 本质 | 当前替代方案痛点 |
|---|---|---|---|
| 1 | **识别购买意图** | 知道客户是否认真评估 | 提案发出后石沉大海 |
| 2 | **映射决策链** | 知道 champion 转发给了谁 | 无法识别多人访问 |
| 3 | **把握跟进时机** | 在客户活跃时联系 | 跟进时机靠猜 |
| 4 | **统一销售资料** | 防止销售用错版本 | 版本混乱 |
| 5 | **CRM 集成** | 自动记录活动 | 手动录入 |
| 6 | **知道说什么** | 基于 engagement 生成建议 | 无数据支撑 |

**典型工作日**：
```
08:30  Slack 收到通知：Acme Corp 查看了价格页
09:00  Dashboard 显示 Acme 被转发给 3 人 → 建议补充安全白皮书
10:00  一键发送 follow-up 邮件
14:00  HubSpot 自动同步今日 engagement 活动
16:00  准备明天的 call，重点讲安全与实施
```

**决策关键词**：Deal Intent、pipeline、决策链、CRM、follow-up、转化率。

## 4. 关键发现交叉分析

### 4.1 三类用户共同痛点

1. **"发出去之后发生了什么"的盲区**
2. **"什么时候跟进"的判断困难**
3. **"材料是否安全"的持续焦虑**
4. **"下一步该做什么"的行动缺失**

### 4.2 三类用户差异化诉求

| 维度 | 创始人 | IR | 销售 |
|---|---|---|---|
| 时间单位 | 融资轮次（月） | 报告周期（季） | 销售周期（周） |
| 核心对象 | 投资人 | LP/联合投资方 | 采购委员会 |
| 成功指标 | 融资进度 | Engagement 率 | Pipeline 转化率 |
| 风险偏好 | 愿为信息透明承担一定风险 | 极低风险，强合规 | 愿为速度牺牲部分安全 |
| UI 语言 | 热度/跟进/轮次 | Engagement/审计/合规 | Intent/Pipeline/Action |

### 4.3 产品机会优先级矩阵

```
高影响 ↑
         ┌─────────────┐
         │ 行动建议系统 │  ← 高影响 + 高差异化
         │ AI 证据定位  │
         └─────────────┘
         ┌─────────────┐
    信号 │ 交易雷达     │  ← 高影响 + 中等差异化
    流   │ 联系人 360°  │
         └─────────────┘
低影响 ↑ ┌─────────────┐
         │ UI 主题切换  │  ← 低影响 + 低差异化
         │ 表格排序     │
         └─────────────┘
         低差异化 → 高差异化
```

## 5. 设计哲学提炼

### 5.1 Signal-First（信号优先）

产品界面应以"今日需要关注什么"为核心，而非"我有哪些文件"。

### 5.2 Actionable Transparency（可行动的透明）

追踪不是目的，让用户基于追踪采取行动才是目的。所有数据必须附带建议。

### 5.3 Friction by Choice（可选的摩擦）

默认低摩擦，用户根据安全需求主动增加摩擦。安全强度与接收方摩擦必须可见平衡。

### 5.4 Trustworthy AI（可信赖的 AI）

AI 输出必须附带证据，证据必须可定位到原文。没有证据的 AI 回答不可信。

### 5.5 Circle-Specific Language（圈层专属语言）

同一底层数据，用三种不同的 UI 语言呈现，降低认知摩擦。

## 6. 风险与假设

### 6.1 关键假设

1. 用户愿意为"行动建议"支付溢价（vs 纯分析工具）
2. 接收方体验足够好，能形成增长飞轮
3. 规则版热度评分在 MVP 阶段足够准确
4. AI 证据定位能建立用户信任

### 6.2 待验证风险

1. **热度评分主观性**：不同圈层的"关键页"定义可能差异大
2. **信号流信息过载**：过多建议可能导致用户麻木
3. **AI 准确性**：mock AI 无法反映真实模型能力
4. **隐私敏感**：接收方可能反感被追踪，需明确隐私说明

---

## 7. 对后续设计的直接输入

1. Dashboard 必须改造为交易雷达
2. 智能链接创建器必须可视化安全/摩擦平衡
3. AI 悬浮助手必须支持证据链跳转
4. 文档详情页必须三位一体
5. 联系人必须 360° 视图
6. 数据室必须模板化
7. 热度评分必须规则先行、圈层独立
